Lesson 2 — Data Structures & Algorithms in JavaScript

Linked Lists, Dictionaries & Recursion

Quick recap: last class we covered arrays — ordered, numbered lockers sitting next to each other in memory. Today we look at two more ways to organize data, plus a new way of thinking about solving problems.

01 — Linked Lists

What if the lockers aren't next to each other?

An array keeps its items in one neat row in memory. A linked list doesn't. Its items can be scattered anywhere in memory — each one just holds a note telling you where to find the next one.

Think of it like... A treasure hunt. You don't get a map of every clue up front. You find clue 1, and it tells you where clue 2 is. Clue 2 tells you where clue 3 is. You can only get to a clue by following the trail from the start — you can't jump straight to clue 5.

Each item in a linked list is called a node. A node has two parts:

Value

The actual data — a number, a name, whatever you're storing.

Next

A pointer (an address) to where the next node lives in memory. The last node points to nothing.

Arrays vs. linked lists

Neither one is "better" — they're just good at different things. This trade-off is the whole point of today's lesson.

TaskArrayLinked list
Jump straight to item #50Fast — just check that lockerSlow — must walk the chain from the start
Add/remove at the very frontSlow — every item shifts overFast — just rewire one pointer

In JavaScript There's no built-in LinkedList type — you build one yourself, usually as a small class for the node (value + next) and a class for the list itself that keeps track of the first node (the "head"). We'll build this together in class.

02 — Dictionaries & Hash Tables

What if you looked things up by name, not by number?

Arrays and linked lists find things using a position — index 0, index 1, and so on. A dictionary (also called a hash table or map) finds things using a key — usually a word — instead of a number.

Think of it like... Your phone's contacts app. You don't look up your friend by "contact #42." You look them up by name, and it instantly shows you their number. Name in, number out. That's a dictionary: a key (the name) mapped to a value (the number).

How does it find things so fast?

Behind the scenes, a dictionary uses a hash function — a little formula that turns a key (like "Amir") into a number. That number tells the computer exactly which memory slot to check, so it doesn't have to search item by item like it might with an array. This is why looking something up in a dictionary is close to instant (O(1)) even with millions of entries.

Dictionaries in JavaScript

JavaScript gives you two easy ways to build one:

// plain object — the simplest way
const contacts = {
  Amir: "555-0134",
  Bella: "555-0198",
  Cy: "555-0177"
};
contacts["Bella"]; // "555-0198"

// Map — a built-in dictionary type, good when keys change a lot
const contactsMap = new Map();
contactsMap.set("Amir", "555-0134");
contactsMap.get("Amir"); // "555-0134"
OperationWhat it meansJavaScript

Array vs. linked list vs. dictionary

Array: fast access by position, slow to insert at the front. Linked list: fast to insert/remove at the front, slow access by position. Dictionary: fast lookup by name/key instead of position at all. Same goal — organizing data — three different trade-offs.

03 — Recursion

What if a function could call itself?

So far, every problem we've solved has used loops — repeat these steps until done. Recursion is a different way to repeat steps: a function that calls itself, with a smaller version of the same problem, until the problem becomes small enough to answer directly.

Every recursive function needs exactly two parts:

Base case

The "tiniest doll" — the simplest version of the problem, answered directly with no further calls. Without this, the function calls itself forever.

Recursive case

The step where the function calls itself again, but with a smaller or simpler version of the problem, moving it closer to the base case.

Each call waits on a smaller call, until the base case answers directly — then the answers pass back up the chain.

Why use recursion at all?

Some problems are naturally defined in terms of smaller versions of themselves — walking a linked list, exploring a tree of folders, or breaking a big task into identical smaller tasks. For those problems, recursion often reads more clearly than a loop, once you're comfortable with it. It's a new way of thinking today — the code itself is something we'll build together in class.
